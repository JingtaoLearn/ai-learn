01Architecture Overview

Anthropic's autonomous-coding demo uses a two-agent pattern to build complete applications across unlimited Claude Code sessions. A thin Python harness orchestrates two specialized agents — an Initializer that bootstraps the project once, and a Coding Agent that implements features across as many sessions as needed.

The harness itself is trivial: a while True loop that creates a fresh ClaudeSDKClient each iteration, selects the right prompt, runs a session, then sleeps 3 seconds and repeats. All complexity lives in the prompts and the filesystem.

02Agent Initialization Flow

Each session starts by instantiating a fresh ClaudeSDKClient via create_client(). The harness detects whether feature_list.json exists to decide which prompt to use — if absent, it's session 1 and the Initializer runs; otherwise the Coding Agent takes over.

Core harness loop — agent.py

async def run_autonomous_agent(project_dir, model, max_iterations=None):
    tests_file = project_dir / "feature_list.json"
    is_first_run = not tests_file.exists()
    iteration = 0
    while True:
        iteration += 1
        if max_iterations and iteration > max_iterations:
            break
        client = create_client(project_dir, model)  # fresh context each time
        if is_first_run:
            prompt = get_initializer_prompt()
            is_first_run = False
        else:
            prompt = get_coding_prompt()
        async with client:
            status, response = await run_agent_session(client, prompt, project_dir)
        await asyncio.sleep(AUTO_CONTINUE_DELAY_SECONDS)  # 3 seconds

The run_agent_session() call drives the Claude Code SDK conversation until the agent completes its session. The harness does not inspect the content of the session — it only checks the exit status returned by the SDK.

Client factory — client.py

def create_client(project_dir, model):
    security_settings = {
        "sandbox": {"enabled": True, "autoAllowBashIfSandboxed": True},
        "permissions": {
            "defaultMode": "acceptEdits",
            "allow": [
                "Read(./**)", "Write(./**)", "Edit(./**)",
                "Glob(./**)", "Grep(./**)", "Bash(*)",
                *PUPPETEER_TOOLS,
            ],
        },
    }
    return ClaudeSDKClient(
        options=ClaudeCodeOptions(
            model=model,
            system_prompt="You are an expert full-stack developer building "
                          "a production-quality web application.",
            allowed_tools=[*BUILTIN_TOOLS, *PUPPETEER_TOOLS],
            mcp_servers={"puppeteer": {"command": "npx",
                                       "args": ["puppeteer-mcp-server"]}},
            hooks={"PreToolUse": [HookMatcher(matcher="Bash",
                                              hooks=[bash_security_hook])]},
            max_turns=1000,
            cwd=str(project_dir.resolve()),
            settings=str(settings_file.resolve()),
        )
    )
Parameter Value Purpose
max_turns 1000 Allow long-running feature implementation without hitting SDK limits
sandbox.enabled True OS-level isolation; Bash commands sandboxed by the Claude Code runtime
defaultMode acceptEdits File edits applied automatically; no human approval required per edit
PreToolUse hook bash_security_hook Command allowlist runs before every Bash tool call
mcp_servers Puppeteer MCP Browser automation for screenshot-based feature verification
cwd project_dir All relative paths resolve inside the project directory

The settings object is written to .claude_settings.json inside the project directory at startup. This means the filesystem restrictions and sandbox config are co-located with the project, not embedded in the harness code.

03Multi-Session State Transfer

Claude Code sessions have no cross-session memory. The harness solves this with a "file as state" pattern: every piece of information the next session needs is written to the filesystem before the current session ends.

Three files carry state across sessions. The Coding Agent's first step is always to read all three before doing any work.

File Written by Read by Content Mutability
feature_list.json Initializer Every session 200 test cases with feature metadata and passes boolean Only the passes field may be changed — all other fields are immutable
claude-progress.txt Every session (end) Every session (start) Free-form progress notes: what was done, what broke, what to do next Overwritten each session with fresh notes
git log Every session (commit step) Every session (bearings step) Commit history as a chronological work record Append-only (normal git)

Immutability constraint on feature_list.json

The Initializer creates feature_list.json once with 200 test cases covering both functional features and style/UX requirements, ordered by priority. The Coding Agent prompt contains an explicit hard constraint: 

IT IS CATASTROPHIC TO REMOVE OR EDIT FEATURES IN FUTURE SESSIONS.
The feature_list.json is the single source of truth for what needs to be built.
In future sessions you may ONLY change the "passes" field of existing entries.

This design means the feature backlog is stable across all N sessions. Work is tracked by flipping "passes": false to "passes": true — a monotonically increasing progress indicator that can never regress below the previous session's count.

Step 1 "Get Your Bearings" — context reconstruction

The Coding Agent begins every session by running exactly these 7 commands to reconstruct the full project context from the filesystem: 

# Step 1: Get Your Bearings
cat app_spec.txt          # Original product requirements
cat feature_list.json     # All 200 features and current pass/fail status
cat claude-progress.txt   # Progress notes from the last session
git log --oneline -20     # Recent commit history
ls -la                    # Current project structure
cat package.json          # Dependencies and scripts (if present)
cat README.md             # Project documentation (if present)

This sequence gives the agent a complete picture: what was requested, what has been built, what was done last session, and what remains. No human context-passing is needed.

04Context Reset Strategy

Every iteration of the harness loop creates a new ClaudeSDKClient instance. This is not a limitation — it is an explicit design choice.

Session N ends
Agent writes progress notes, commits, exits cleanly
▼ 3-second delay (AUTO_CONTINUE_DELAY_SECONDS)
create_client() called again
Fresh ClaudeSDKClient — no session ID, no --resume, no conversation history
▼
Session N+1 begins
Empty context window — agent reads filesystem to reconstruct state
Property Design Choice Why
Session continuity None — no --resume, no conversation ID Context window exhaustion is guaranteed on long tasks; proactive reset avoids degraded performance from a bloated context
State carrier Filesystem only Files are durable, inspectable, and independent of the Claude API
Reset trigger After every session, unconditionally Simpler than detecting context pressure; predictable cost model
Inter-session delay 3 seconds Avoids API rate-limiting between rapid session starts

Context Reconstruction Pattern

The Coding Agent's prompt explicitly tells it that it is in a fresh context window and that it must reconstruct its understanding before doing any work: 

YOUR ROLE - CODING AGENT
This is a FRESH context window - you have no memory of previous sessions.
Your entire understanding of the project state must come from reading files.

Step 1 (MANDATORY): Read app_spec.txt, feature_list.json, claude-progress.txt,
                    and git log before doing anything else.

This "prompt-driven recovery" pattern means the agent doesn't depend on implicit session state. The harness never needs to serialize or inject prior conversation history — the agent self-orients by design.

05Prompt Design

The two prompts are the core intellectual content of the system. The harness is ~60 lines; each prompt is hundreds of lines of precise, step-by-step instructions.

Initializer Prompt

Runs exactly once (Session 1). Produces all the scaffolding that every subsequent session depends on.

Task Output Notes
1. Read app_spec.txt Understanding of product requirements Provided by the user in the project directory
2. Create feature_list.json 200 test cases, priority-ordered Mixed: functional + style, step counts 2–5 and 10+, priority-ordered
3. Create init.sh Startup script for dev servers Used by every coding session to start the app before testing
4. Git init + project structure Initial commit + directory layout Baseline commit so git log starts clean
5. End session cleanly Commit + claude-progress.txt Progress notes document what was set up and what to build first
▶ Initializer prompt excerpt 
YOUR ROLE - INITIALIZER AGENT (Session 1 of Many)

You are setting up a new project. Future sessions will implement features
one at a time using the scaffold you create now.

TASKS (in order):
1. Read app_spec.txt to understand what needs to be built
2. Create feature_list.json with EXACTLY 200 test cases:
   - Mix of functional features and style/UX requirements
   - Include both quick tasks (2-5 steps) and complex ones (10+ steps)
   - Priority-ordered: most important features first
   - Every entry has: id, title, description, category, passes (false)
3. Create init.sh — script to start all dev servers
4. Initialize git repository and create initial project structure
5. Before session ends:
   - Commit everything with a clear message
   - Create claude-progress.txt documenting what was set up

IT IS CATASTROPHIC TO REMOVE OR EDIT FEATURES IN FUTURE SESSIONS.
The feature_list.json you create now is the permanent feature backlog.

Coding Prompt — 10-Step Workflow

Runs for every session from Session 2 onward. The 10 steps encode three key principles: reconstruct before acting, verify before building, and one feature at a time.

Step Action Key Constraint
1 Get Your Bearings Read all 4 state files before anything else
2 Start Servers Run ./init.sh to bring up the dev environment
3 Regression Test (MANDATORY) Verify 1–2 already-passing features before writing any new code
4 Choose ONE Feature Pick the highest-priority unimplemented feature from feature_list.json
5 Implement Write code, one feature only
6 Browser Verify (Puppeteer) Take screenshots proving the feature works; no skipping this step
7 Update feature_list.json ONLY change the "passes" field of the implemented feature
8 Commit One commit per feature with a descriptive message
9 Update claude-progress.txt Document what was done, what broke, and what to tackle next session
10 End Session Cleanly No dangling processes, no uncommitted changes

"Regression before progress" principle

Step 3 is explicitly marked MANDATORY in the prompt. Before implementing any new feature, the agent must verify that previously-passing features still pass. This provides a regression safety net across context resets — if a prior session introduced a bug and the agent failed to catch it, the next session will detect it during Step 3 before compounding the problem.

▶ Coding prompt excerpt 
YOUR ROLE - CODING AGENT
This is a FRESH context window - you have no memory of previous sessions.

Step 1: Get Your Bearings
  cat app_spec.txt
  cat feature_list.json
  cat claude-progress.txt
  git log --oneline -20

Step 2: Start Servers
  ./init.sh

Step 3: Verification Test — MANDATORY: test 1-2 passing features BEFORE new work
  Use Puppeteer to verify existing features still work.
  If regression found, fix it before proceeding.

Step 4: Choose ONE Feature to implement
  Pick the highest-priority feature where passes=false

Step 5: Implement the feature

Step 6: Verify with Browser Automation (Puppeteer screenshots)
  Must see visual proof the feature works before marking it as passing.

Step 7: Update feature_list.json (ONLY change "passes" field to true)

Step 8: Commit with a descriptive message

Step 9: Update claude-progress.txt with:
  - What feature was implemented
  - Any issues encountered
  - What to tackle next session

Step 10: End Session Cleanly — kill dev servers, verify clean git state

06Security Model

The harness applies defense-in-depth with three independent security layers. Each layer would need to fail independently for an unsafe command to execute.

Layer 1: OS Sandbox
sandbox.enabled = true in .claude_settings.json — Claude Code runtime OS isolation
▼
Layer 2: Filesystem Restrictions
Read(./**), Write(./**), Edit(./**) — all paths relative to project dir only
▼
Layer 3: Bash Command Allowlist (PreToolUse Hook)
bash_security_hook() in security.py — blocks any command not in ALLOWED_COMMANDS

ALLOWED_COMMANDS — security.py

ALLOWED_COMMANDS = {
    # File inspection
    "ls", "cat", "head", "tail", "wc", "grep",
    # File manipulation
    "cp", "mkdir", "chmod",
    # Navigation
    "pwd",
    # Runtime
    "npm", "node",
    # Version control
    "git",
    # Process management
    "ps", "lsof", "sleep", "pkill",
    # Project startup
    "init.sh",
}

async def bash_security_hook(input_data, tool_use_id=None, context=None):
    command = input_data.get("tool_input", {}).get("command", "")
    commands = extract_commands(command)
    for cmd in commands:
        if cmd not in ALLOWED_COMMANDS:
            return {"decision": "block", "reason": f"Command '{cmd}' not allowed"}
        # Extra validation for sensitive commands:
        # pkill  → only dev process names allowed (node, npm, etc.)
        # chmod  → only +x flag allowed (no permission widening)
        # init.sh → only ./init.sh allowed (no path traversal)
    return {}  # allow
Sensitive Command Extra Validation Blocked Example
pkill Only dev process names (node, npm, vite, etc.) are allowed targets pkill -9 sshd → blocked
chmod Only +x flag allowed; numeric modes blocked chmod 777 file → blocked
init.sh Only ./init.sh form; path traversal blocked ../init.sh → blocked

The .claude_settings.json file is written to the project directory at harness startup. This keeps the security configuration version-controlled alongside the project rather than embedded in a global config.

07Key Architectural Insights

The autonomous-coding demo distills several reusable patterns for building reliable long-running AI coding agents.

File as State

No agent memory, no database, no session continuity. Every piece of state lives in files the agent writes and reads. The filesystem is the only cross-session contract.

Why it works: Files are durable, inspectable by humans, and independent of the Claude API. Debugging a stuck agent means reading claude-progress.txt — no special tooling needed.

Stateless Worker

The Python harness is intentionally trivial. It creates clients, selects prompts, and loops. All intelligence is in the prompts — the harness has zero domain knowledge.

Why it works: Simple harness = easy to test, reason about, and extend. Adding a new agent type means adding a new prompt file, not modifying the harness.

Immutable + Monotonic

feature_list.json is immutable except for the passes field, which only ever changes from false to true. Progress is monotonically increasing.

Why it works: The agent can never accidentally delete or reorder the backlog. Progress is always visible and measurable across any number of sessions.

Regression Before Progress

Every session begins with a mandatory regression check before new work. The agent must verify existing features still pass using Puppeteer before implementing anything.

Why it works: Context resets introduce the risk of silent regressions. Step 3 turns this from a latent bug into an immediate, session-local problem that must be fixed.

Defense in Depth

Three independent security layers: OS sandbox, filesystem path restrictions, and a Bash command allowlist hook. Each can block malicious activity independently.

Why it works: A single allowlist can be bypassed by creative command construction. Three layers means three independent failure modes must all align for a breach.

Proactive Context Reset

Context is reset after every session, not when it's exhausted. The agent reconstructs from files at the start of each session rather than relying on accumulated history.

Why it works: Degraded performance from a bloated context window is worse than the overhead of reconstructing from files. The cost of reconstruction is predictable; the cost of degraded context is not.
